  <name>Task 1: Create progression SQL migration with table, RPC functions, and RLS policies</name>
  <files>supabase/migrations/00005_progression_suggestions.sql</files>
  <action>
Create SQL migration file `supabase/migrations/00005_progression_suggestions.sql` with:

1. **progression_status enum**: `CREATE TYPE progression_status AS ENUM ('pending', 'accepted', 'dismissed');`

2. **progression_suggestions table** with columns:
   - `id UUID DEFAULT gen_random_uuid() PRIMARY KEY`
   - `client_id UUID NOT NULL REFERENCES profiles(id)`
   - `exercise_id UUID NOT NULL REFERENCES exercises(id)`
   - `plan_exercise_id UUID NOT NULL REFERENCES plan_exercises(id)`
   - `current_weight_kg DECIMAL(6,2) NOT NULL`
   - `suggested_weight_kg DECIMAL(6,2) NOT NULL`
   - `triggered_by_session_id UUID NOT NULL REFERENCES workout_sessions(id)`
   - `status progression_status NOT NULL DEFAULT 'pending'`
   - `resolved_at TIMESTAMPTZ`
   - `resolved_by UUID REFERENCES profiles(id)`
   - `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`

3. **Partial index** for fast pending lookup: `CREATE INDEX idx_progression_pending ON progression_suggestions(client_id, status) WHERE status = 'pending';`

4. **RLS policies** (enable RLS first, then granular per-operation policies following project convention of separate policies per operation, never FOR ALL):
   - SELECT: Client can see own suggestions (`client_id = auth.uid()`) OR user is trainer (`is_trainer()` helper from Phase 2 migration)
   - INSERT: Allow authenticated users where `client_id = auth.uid()` (RPC runs as invoker in client context)
   - UPDATE: Client can update own (`client_id = auth.uid()`) OR trainer (`is_trainer()`), WITH CHECK constraining status to 'accepted' or 'dismissed'
   - No DELETE policy (suggestions are never deleted, only status-transitioned)

5. **`check_progression_eligibility(p_session_id UUID)` RPC function** (SECURITY INVOKER, returns SETOF progression_suggestions):
   - Query workout_sets for the given session where `reps >= 15`
   - Join to plan_exercises to get prescribed_weight_kg
   - **Filter for sets at prescribed weight only** (per research pitfall 1 -- warm-up filtering): `AND ws_sets.weight_kg >= pe.prescribed_weight_kg` (only evaluate sets at or above prescribed weight, not warm-up sets at lighter weight)
   - For each qualifying exercise, check if a pending suggestion already exists for same `(client_id, exercise_id, current_weight_kg)` -- skip if one exists
   - Also check if a dismissed suggestion exists for same `(client_id, exercise_id, current_weight_kg)` created within the last 3 sessions (re-suggestion cooldown per user decision: re-suggest after 3 sessions). Count sessions by querying completed workout_sessions for that client after the dismissed suggestion's created_at. Only re-suggest if 3+ sessions have been completed since dismissal.
   - Use per-exercise increment: Check if exercises table has a `progression_increment_kg` column. Since it does not exist yet, add an ALTER TABLE to add `progression_increment_kg DECIMAL(4,2) DEFAULT NULL` to the exercises table. The suggested weight = `current_weight_kg + COALESCE(e.progression_increment_kg, 2.5)` (default 2.5kg per user decision)
   - INSERT new suggestion and RETURN QUERY the inserted row

6. **`accept_progression_suggestion(p_suggestion_id UUID)` RPC function** (SECURITY INVOKER, returns void):
   - SELECT the suggestion with FOR UPDATE lock where `status = 'pending'`
   - RAISE EXCEPTION if not found or not pending
   - **Stale check** (per research pitfall 3): Verify `plan_exercises.prescribed_weight_kg` still matches `progression_suggestions.current_weight_kg`. If weight has changed (stale), auto-dismiss instead of accepting.
   - UPDATE plan_exercises SET `prescribed_weight_kg = suggestion.suggested_weight_kg`
   - UPDATE progression_suggestions SET `status = 'accepted'`, `resolved_at = now()`, `resolved_by = auth.uid()`

7. **`dismiss_progression_suggestion(p_suggestion_id UUID)` RPC function** (SECURITY INVOKER, returns void):
   - SELECT suggestion with FOR UPDATE lock where `status = 'pending'`
   - RAISE EXCEPTION if not found or not pending
   - UPDATE progression_suggestions SET `status = 'dismissed'`, `resolved_at = now()`, `resolved_by = auth.uid()`

Note: The exercises table ALTER TABLE for `progression_increment_kg` is included in this migration because the detection function references it. Default NULL means 2.5kg is used (COALESCE pattern). Trainer can optionally set per-exercise increments via the exercise form in a future enhancement.
