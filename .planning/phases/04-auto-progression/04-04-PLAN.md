<objective>
Fix the two compounding bugs that prevent progression suggestions from ever appearing on the post-workout Session Summary and trainer client detail view.

Purpose: UAT tests 1 and 6 failed because (a) the fire-and-forget RPC call silently swallows all errors -- supabase.rpc() never rejects, so .catch() is dead code and .then() never inspects the error field -- and (b) setIsCompleted(true) fires before the RPC completes, so even if suggestions were created, the initial query returns empty and the user may navigate away before refetch.

Output: Properly error-handled RPC integration that logs failures, and a race-condition-free post-workout flow where suggestions appear immediately when the summary mounts.
</objective>

The key insight is that supabase.rpc() NEVER rejects -- it always resolves with {data, error}. The current .catch() is dead code and .then() ignores the error field.

       - Awaits `supabase.rpc("check_progression_eligibility", { p_session_id: data.id })`
       - Checks the `error` field: `const { error: rpcError } = await supabase.rpc(...)`
       - If `rpcError`, logs to console: `console.error("Progression detection failed:", rpcError.message)` -- do NOT toast (detection failure should not alarm the user)
       - If no error, invalidates `progressionKeys.pending(data.clientId)`
    3. Make the onSuccess callback async so it can await the RPC
    4. Keep session-related invalidations and the completion toast BEFORE the RPC call (session completion UX must remain instant)

    The updated onSuccess should look like:
    ```
    onSuccess: async (data) => {
      // Immediate session completion feedback (no delay)
      queryClient.invalidateQueries({ queryKey: workoutKeys.sessions(data.clientId) });
      queryClient.invalidateQueries({ queryKey: dashboardKeys.clients() });
      toast.success(t("workouts.completed", "Workout completed!"));

      // Await progression detection (error-checked, non-blocking to user)
      const { error: rpcError } = await supabase.rpc(
        "check_progression_eligibility",
        { p_session_id: data.id }
      );
      if (rpcError) {
        console.error("Progression detection failed:", rpcError.message);
      }
      queryClient.invalidateQueries({
        queryKey: progressionKeys.pending(data.clientId),
      });
    },
    ```

    **workout-session.tsx -- Fix race condition:**

    The problem: `completeSession.mutate(...)` has an inline `onSuccess: () => setIsCompleted(true)` that fires as soon as the DB update completes, but BEFORE the progression RPC in the mutation-level onSuccess finishes. This means SessionSummary mounts and queries for suggestions while the RPC is still in-flight.

    Fix by making handleFinishWorkout use mutateAsync instead of mutate, so the mutation-level onSuccess (which now awaits the RPC) completes fully before setIsCompleted fires:

    1. Change `completeSession.mutate(...)` to `await completeSession.mutateAsync(...)` in handleFinishWorkout
    2. Make handleFinishWorkout async
    3. Wrap in try/catch -- on error, do nothing (the mutation's onError already toasts)
    4. Move `setIsCompleted(true)` AFTER the await (not in an inline onSuccess)
    5. Remove the inline `onSuccess` callback from the mutate options

    The updated handleFinishWorkout should look like:
    ```
    const handleFinishWorkout = useCallback(async () => {
      if (!activeSession || !clientId) return;
      try {
        await completeSession.mutateAsync({ sessionId: activeSession.id, clientId });
        setIsCompleted(true);
      } catch {
        // Mutation-level onError already shows toast
      }
    }, [activeSession, clientId, completeSession]);
    ```

    This ensures:
    - Session is marked complete in DB
    - Toast shows immediately ("Workout completed!")
    - RPC runs and creates suggestions
    - Progression cache is invalidated
    - THEN setIsCompleted(true) fires
    - SessionSummary mounts and usePendingSuggestions fetches -- suggestions are already in DB

Remove the entire function but keep the progressionKeys factory and the other 3 hooks (usePendingSuggestions, useAcceptSuggestion, useDismissSuggestion).
