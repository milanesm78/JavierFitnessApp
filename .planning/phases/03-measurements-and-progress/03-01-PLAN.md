**SQL Migration (supabase/migrations/00003_body_measurements.sql):**
Create body_measurements table with flat column schema (NOT EAV pattern -- per research anti-pattern warning). Columns:
- id UUID PK DEFAULT gen_random_uuid()
- client_id UUID NOT NULL REFERENCES profiles(id)
- measured_at TIMESTAMPTZ NOT NULL DEFAULT now()
- weight DECIMAL(5,1) NOT NULL, height DECIMAL(4,1) NOT NULL
- 6 skin fold columns (DECIMAL(4,1), nullable): skinfold_triceps, skinfold_subscapular, skinfold_suprailiac, skinfold_abdominal, skinfold_thigh, skinfold_calf
- 3 bone diameter columns (DECIMAL(3,1), nullable): diameter_humeral, diameter_femoral, diameter_bistyloidal
- 7 circumference columns (DECIMAL(4,1), nullable): circ_arm_relaxed, circ_arm_flexed, circ_chest, circ_waist, circ_hip, circ_thigh, circ_calf
- notes TEXT nullable
- created_at TIMESTAMPTZ NOT NULL DEFAULT now()
- updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
- CHECK constraints for weight (20-300) and height (100-250)

Create index: idx_measurements_client_date ON body_measurements(client_id, measured_at DESC).

RLS policies -- follow Phase 1/2 convention of separate policies per operation, using is_trainer() helper:
- "Clients read own measurements" FOR SELECT USING (auth.uid() = client_id)
- "Trainer reads all measurements" FOR SELECT USING (is_trainer())
- "Trainer inserts measurements" FOR INSERT WITH CHECK (is_trainer())
- "Trainer updates measurements" FOR UPDATE USING (is_trainer())
- "Trainer deletes measurements" FOR DELETE USING (is_trainer())

Note: Only the trainer enters measurements (real-world workflow: Javier measures clients in person). Client can only read their own data.

Also create RPC function get_strength_progress(p_client_id UUID, p_exercise_id UUID, p_from_date DATE DEFAULT (now() - INTERVAL '12 months')::date, p_to_date DATE DEFAULT now()::date) that returns TABLE(date DATE, max_weight DECIMAL). Query joins workout_sets -> workout_sessions, filters by client_id, exercise_id, completed_at range, groups by date, returns MAX(weight_kg). Use SECURITY DEFINER. This function serves Plan 02's strength charts but is created here with the migration.

**Zod schema (src/features/measurements/schemas.ts):**
Use Zod v4 (import from 'zod') matching existing project pattern. Create helper functions:
- optionalMeasurement(min, max) returns z.number().min(min).max(max).nullable().optional() with transform that converts undefined/NaN to null
- requiredMeasurement(min, max) returns z.number().min(min).max(max)

Build measurementSchema z.object with all fields. Use ranges from research:
- weight: 20-300 (required), height: 100-250 (required)
- All skin folds: 2-60 (optional/nullable)
- Bone diameters: humeral 4-10, femoral 6-14, bistyloidal 3-8 (optional/nullable)
- Circumferences: arm_relaxed 15-60, arm_flexed 15-65, chest 60-150, waist 40-150, hip 50-160, thigh 30-90, calf 20-60 (optional/nullable)
- notes: z.string().optional()
- measured_at: z.string() (ISO date string)

Export type MeasurementFormValues = z.infer<typeof measurementSchema>.

**TanStack Query hooks (src/features/measurements/hooks/useMeasurements.ts):**
Create measurementKeys query key factory following exerciseKeys/planKeys convention:
- measurementKeys.all, measurementKeys.lists(), measurementKeys.list(clientId), measurementKeys.latest(clientId)

Hooks:
- useClientMeasurements(clientId): Fetches all measurements for a client ordered by measured_at DESC. Returns { data, isPending, error }.
- useLatestMeasurement(clientId): Fetches the single most recent measurement for a client (used to show previous values as placeholder reference in form). Uses .maybeSingle().
- useCreateMeasurement(): Mutation that inserts a row into body_measurements. Accepts { clientId, data: MeasurementFormValues }. Converts form data to DB row (normalize decimals using existing normalizeDecimal from @/lib/utils if needed, handle null optionals). Invalidates measurementKeys.list(clientId) on success. Shows success toast via sonner.

The hooks must handle the decimal normalization pattern established in Phase 1: form data may contain string-like numeric values from DecimalInput, so mutation layer converts to DB-native types before insert.
