  <name>Task 2: Create Supabase database migration with complete Phase 1 schema</name>
  <files>
    supabase/migrations/00001_initial_schema.sql

  <action>
    Create `supabase/migrations/00001_initial_schema.sql` containing the complete Phase 1 database schema. This file is run manually against Supabase (via SQL Editor or `supabase db push`) -- it is NOT auto-executed.

    The migration must include, in this order:

    1. **User role enum:**
       `create type public.user_role as enum ('trainer', 'client');`

    2. **Profiles table** (extends auth.users):
       - `id uuid references auth.users on delete cascade primary key`
       - `email text not null`
       - `full_name text`
       - `role user_role not null default 'client'`
       - `is_active boolean not null default false` (trainer must activate clients)
       - `preferred_language text not null default 'es'`
       - `theme text not null default 'dark'` (per user decision: dark/light theme support)
       - `created_at timestamptz not null default now()`
       - `updated_at timestamptz not null default now()`

    3. **User roles table** (for Custom Access Token Hook):
       - `id bigint generated by default as identity primary key`
       - `user_id uuid references auth.users on delete cascade not null`
       - `role user_role not null`
       - `unique (user_id, role)`

    4. **Exercises table:**
       - `id uuid default gen_random_uuid() primary key`
       - `name text not null`
       - `youtube_url text not null`
       - `created_by uuid references auth.users not null default auth.uid()`
       - `created_at timestamptz not null default now()`
       - `updated_at timestamptz not null default now()`

    5. **Indexes** for RLS performance:
       - `idx_profiles_role on public.profiles(role)`
       - `idx_profiles_is_active on public.profiles(is_active)`
       - `idx_user_roles_user_id on public.user_roles(user_id)`
       - `idx_exercises_created_by on public.exercises(created_by)`

    6. **Enable RLS** on ALL tables: profiles, user_roles, exercises

    7. **RLS Policies for profiles:**
       - Users can view own profile: `using (id = (select auth.uid()))`
       - Trainer can view all profiles: `using ((select auth.jwt()->>'user_role') = 'trainer')`
       - Trainer can update any profile (for activation): `using/with check ((select auth.jwt()->>'user_role') = 'trainer')`
       - Users can update own profile (name, language, theme): `using/with check (id = (select auth.uid()))`
       - See RESEARCH.md "RLS Policies for Profiles Table"

    8. **RLS Policies for user_roles:**
       - Only supabase_auth_admin can read (for the Custom Access Token Hook)
       - See RESEARCH.md "Custom Access Token Hook" grant section

    9. **RLS Policies for exercises:**
       - Trainer can SELECT, INSERT, UPDATE, DELETE (separate policies, NOT `FOR ALL`)
       - Active clients can SELECT (needed for Phase 2 workout view)
       - See RESEARCH.md "RLS Policies for Exercises Table"

    10. **Custom Access Token Hook function** (`public.custom_access_token_hook`):
        - Reads user role from `user_roles` table
        - Injects `user_role` claim into JWT
        - Grant permissions to `supabase_auth_admin`
        - Revoke from authenticated, anon, public
        - See RESEARCH.md "Custom Access Token Hook" code example

    11. **Profile auto-creation trigger** (`handle_new_user`):
        - Creates profile row when user signs up
        - Also inserts into user_roles table
        - Reads `full_name` and `role` from `raw_user_meta_data` (passed during signUp)
        - Trainer is auto-activated; clients default to inactive
        - Uses `security definer set search_path = ''` for security
        - See RESEARCH.md "Profile Auto-Creation Trigger"

    12. **Updated_at trigger:**
        - Create a reusable function `update_updated_at_column()` that sets `updated_at = now()`
        - Attach to `profiles` and `exercises` tables on UPDATE

    Add clear SQL comments explaining each section.
  </action>
