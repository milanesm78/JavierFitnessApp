  <action>
    1. Create `src/features/auth/types.ts`:
       - Export `UserRole = 'trainer' | 'client'`
       - Export `AuthState` interface: `session: Session | null`, `user: User | null`, `userRole: UserRole | null`, `isActive: boolean | null`, `isLoading: boolean`

    2. Create `src/features/auth/context/AuthProvider.tsx`:
       - React Context providing: session, user, userRole, isActive, isLoading, signIn, signUp, signOut
       - On mount: call `supabase.auth.getSession()` to restore session, then set up `onAuthStateChange` listener
       - **Session persistence (AUTH-04):** Supabase stores session in localStorage by default (`persistSession: true`). No custom code needed -- just verify by NOT disabling it.
       - **Role extraction (AUTH-02):** When session exists, decode JWT from `session.access_token` using `JSON.parse(atob(token.split('.')[1]))` to get `user_role` claim. Do NOT use a separate library for this.
       - **Activation status (AUTH-03):** After getting session, query `supabase.from('profiles').select('is_active').eq('id', user.id).single()` to get activation status. Store in context as `isActive`.
       - **isLoading starts as `true`**, set to `false` only after initial session check + profile query complete. This prevents flash of login page (Pitfall 4 from RESEARCH.md).
       - `signIn(email, password)`: Call `supabase.auth.signInWithPassword()`. Handle error by throwing with message.
       - `signUp(email, password, fullName, role)`: Call `supabase.auth.signUp()` with `options.data: { full_name: fullName, role }`. The DB trigger creates profile and user_role records.
       - `signOut()`: Call `supabase.auth.signOut()`.
       - Export `useAuth` hook that reads context (throws if used outside provider).
       - See RESEARCH.md "Pattern 1: Supabase Auth with React Context"

    3. Create `src/features/auth/hooks/useAuth.ts`:
       - Re-export `useAuth` from AuthProvider for cleaner import paths

    4. Create `src/features/auth/components/LoginForm.tsx`:
       - Form with email and password inputs using shadcn/ui Input and Label
       - Submit button using shadcn/ui Button
       - All text uses `useTranslation()` -- no hardcoded strings
       - Loading state on submit button (disable + spinner)
       - Error display using shadcn/ui toast (via sonner)
       - Link to register page
       - Mobile-first layout: full width on mobile, max-w-sm centered on desktop

    5. Create `src/features/auth/components/RegisterForm.tsx`:
       - Form with: full name, email, password, role selection (trainer/client radio or select)
       - **Role selection (AUTH-02):** User picks "Trainer" or "Client" during registration. Per user decision, this is how roles are assigned. (Research recommended seeding trainer, but user decision takes precedence -- the CONTEXT.md says "Client self-registers" implying role choice is available)
       - All text translated via i18next
       - Validation: email format, password minimum 6 chars, name required
       - On success: show success message. If client role, inform they need trainer activation.
       - Link to login page
       - Same mobile-first layout as LoginForm

    6. Create `src/pages/LoginPage.tsx` and `src/pages/RegisterPage.tsx`:
       - Minimal page wrappers that center the respective form components
       - Include LanguageToggle in top-right corner (login/register are public pages, need language access)
       - App branding/logo at top

    7. Update `src/main.tsx`:
       - Wrap app with `AuthProvider` INSIDE `BrowserRouter` (auth needs routing context for redirects)
       - Provider order: React.StrictMode > QueryClientProvider > BrowserRouter > AuthProvider > App
